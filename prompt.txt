High-Level Project
------------------
This is a School Management System for Global Knowledge School. The project has a frontend built with Next.js and a backend built with Laravel. The frontend is intended to be deployed on Vercel. The backend is intended to be deployed on Render. The database is PostgreSQL hosted through Supabase and can be managed using pgAdmin/Supabase tools.

Important deployment rule: the frontend must never connect directly to Supabase/PostgreSQL. The frontend talks only to the Laravel API through HTTP endpoints. The Laravel backend handles database access.

Frontend Environment Variables
------------------------------
The frontend API helper reads:

- NEXT_PUBLIC_API_URL

This must be set in Vercel and local .env files to the Laravel/Render API base URL. Example shape:

NEXT_PUBLIC_API_URL=https://your-render-backend-url.com/api

The helper trims trailing slashes, so both URLs with or without a final slash are acceptable. Keep API endpoint calls relative, for example apiRequest("/students").

API Helper
----------
File: frontend/lib/api.js

All normal frontend API calls should use apiRequest(endpoint, method = "GET", body = null, token = null).

Behavior:
- Reads API base URL from NEXT_PUBLIC_API_URL.
- Removes trailing slashes from API base.
- Sends Accept: application/json.
- Adds Authorization: Bearer <token> only when a token is passed.
- Sends JSON by default when body is not FormData.
- Sends FormData directly without Content-Type so the browser sets multipart boundaries.
- Parses JSON response when possible.
- Handles Laravel-style validation errors from data.errors by throwing the first validation message.
- Throws data.message or "Request failed" for failed non-validation responses.

Important: Some pages still use direct fetch for multipart upload. Prefer apiRequest for new code unless the existing page pattern clearly requires otherwise.

Auth Storage Pattern
--------------------
The active pages do not consistently use token-based auth. Instead, login pages store the returned role object in localStorage:

- Admin login stores localStorage["admin"] = JSON.stringify(data.admin)
- Teacher login stores localStorage["teacher"] = JSON.stringify(data.teacher)
- Student login stores localStorage["student"] = JSON.stringify(data.student)

The sidebar layout for each role checks the matching localStorage key and redirects to that role's login page when missing or invalid.

There is also frontend/lib/auth.js with token/role helpers:
- getToken()
- getRole()
- clearAuth()
- fetchCurrentUser(role) -> calls /{role}/me with Bearer token

However, most current role pages use the role-specific localStorage objects instead of token/role. Before changing auth, inspect backend login responses and keep frontend/backend behavior aligned.

Frontend Coding Rules For Future AI
-----------------------------------
- Keep the App Router route structure: app/admin, app/teacher, app/student.
- Prefer apiRequest from lib/api.js for new API calls.
- Keep NEXT_PUBLIC_API_URL as the single frontend API base config.
- Do not hard-code localhost or deployment URLs in components.
- Do not connect directly to Supabase from the frontend.
- Keep Laravel API field names exactly aligned with backend request validation.
- Use FormData for file uploads such as student/teacher pictures.
- Keep localStorage key names consistent unless also updating every dependent page/layout:
  - admin
  - teacher
  - student
- Preserve role separation. Admin pages should not depend on teacher/student localStorage and vice versa.
- Use Tailwind CSS utility classes, matching existing visual style.
- Prefer lucide-react icons if adding new dashboard/sidebar actions.
- Be careful when changing notices because both homepage and role portals depend on the /notices response shape.
- For deployment on Vercel, ensure the frontend builds without TypeScript/ESLint blockers.

Most Important User Flows
-------------------------
1. Public visitor opens homepage, sees school info, slider, portal links, and homepage notices.
2. Admin logs in, adds students/teachers, manages exam routines, creates notices, and approves/rejects marks.
3. Teacher signs up/logs in, takes attendance, views attendance history, views routines, enters marks, and reads notices.
4. Student signs up/logs in, views dashboard, exam routine, attendance, approved results, and notices.

Rules
